# decompose_input 设计反思

## 一、背景

Phase 2 新增了 `decompose_input()` 功能，目的是处理用户在一条消息里混合多类问题的场景。

设计初衷：
> "我发烧38度，布洛芬和阿司匹林能一起吃吗"
> → 拆分为 ["我发烧38度", "布洛芬和阿司匹林能一起吃吗"]
> → 分别路由到 TriageAgent 和 MedicationAgent

---

## 二、实际测试中发现的问题

### 问题 1：正常症状描述被错误拆分

**输入：**
```
我右上腹痛了两天，吃东西后会加重
```

**期望 decompose 结果：**
```
['我右上腹痛了两天，吃东西后会加重']   # 一个整体
```

**实际 decompose 结果（修复前）：**
```
['我右上腹痛了两天', '吃东西后会加重']  # 被错误拆成两个子问题
```

**后果：**
- 两个子问题各自触发一次完整分诊流程
- 用户看到两条建议，内容重复且逻辑混乱

**根因：**
prompt 只说"逗号分隔的多个问题就拆分"，没有区分"症状的多个维度描述"和"真正独立的两个问题"。

**修复：**
改进 prompt，明确"同一症状的不同描述维度（部位、时间、诱因）不要拆分，只有完全不同类型的问题才拆"。修复后测试通过。

---

### 问题 2：追问与用药回答顺序割裂

**输入：**
```
我发烧38度，布洛芬和阿司匹林能一起吃吗
```

**decompose 结果（正确）：**
```
['我发烧38度', '布洛芬和阿司匹林能一起吃吗']
```

**期望行为：**
- 发烧症状信息不足 → 追问部位
- 等用户回答后继续分诊
- 用药咨询同时或稍后回答

**实际行为（第一次修复前）：**
```
Medi: 请问您感觉不舒服的部位是哪里？
# 追问发出后，for 循环继续执行，不等用户回答
Medi: 布洛芬和阿司匹林不建议同时服用...
# 用药回答紧接在追问后输出，顺序割裂
```

**根因：**
`produce()` 用 for 循环依次处理子问题，追问通过 bus 发出事件后**不阻塞**，循环立即执行下一个子问题。追问和用药回答在同一轮内全部输出，用户还没回答就收到了两条不相关的内容。

**第一次修复（break）：**
症状触发追问后 break，跳过后续子问题。

**后果：**
用药咨询子问题被完全丢弃，用户问的药能不能一起吃没有得到回答。

**第二次修复（symptom_needs_followup 标志位）：**
症状触发追问后设置标志，用药咨询不受影响照常执行。

**实际效果：**
```
Medi: 请问您感觉不舒服的部位是哪里？
Medi: 布洛芬和阿司匹林不建议同时服用...
```

技术上"正确"了，但**用户体验仍然割裂**：
- 用户心里优先想知道药能不能一起吃
- 系统先追问部位，然后突然插入用药回答
- 两条回复之间没有逻辑关联，体验跳跃

---

### 问题 3：两个完整问题同时出现时无法正确处理

**输入：**
```
我昨天胸部右上方有些抽痛该去哪里就诊？我能吃家里有的xxx药吗？
```

**分析：**

这句话包含两个**各自信息完整**的独立问题：
- `"我昨天胸部右上方有些抽痛该去哪里就诊"` — 有部位（胸部右上方）、时间（昨天）、性质（抽痛），信息足够直接分诊
- `"我能吃家里有的xxx药吗"` — 完整的用药咨询

**无论主从如何，都无解：**

- 主意图判成 SYMPTOM → 直接分诊，用药问题丢掉
- 主意图判成 MEDICATION → 回答用药，第二轮补充分诊（行得通）
- 但 `classify_intent` 拿到整句话时判断结果不稳定，无法保证每次都选同一个主意图

**进一步探索：串行处理 + 二轮补充**

曾考虑方案：主意图处理完后，做第二轮判断"输入中是否还有未处理的问题"，有则补充处理。

但这个方案在边界场景下仍然有问题：
- 两个问题都完整时，主从判断本身就不稳定
- 二轮判断又引入了额外的 LLM 调用和新的边界情况
- 本质上还是在绕回 `decompose_input` 的并行处理问题，只是换了执行顺序

**结论：CLI 架构下无法优雅解决**

真正的并行处理需要 UI 层支持——两个问题同时展示两个独立的回答块。CLI 的线性输出无法承载这种交互模式。

---

## 三、根本问题

`decompose_input` 的核心假设是：**用户会在一条消息里混合完全不同类型的问题，且两类问题应该并行处理。**

这个假设在技术上可以实现，但在医疗对话场景下存在根本性缺陷：

1. **混合输入是边缘场景，频率很低。** 大多数用户不会在一句话里同时描述症状和咨询用药。

2. **并行处理破坏对话的连贯性。** 医疗对话需要一个清晰的主线，同时推进两条线索会让用户困惑。

3. **拆分粒度难以把握。** 症状描述天然用逗号连接多个维度，LLM 判断"是否应该拆分"本身就不稳定。

4. **追问的异步性与顺序执行的矛盾无解。** 追问需要等用户回答才能继续，但 for 循环无法在中途暂停等待用户输入。

---

4. **CLI 的线性输出无法承载并行回答。** 真正的并行处理需要前端 UI 支持（两个独立回答块），CLI 的顺序输出结构天然不适合这种交互模式。

---

## 四、反思与结论

### 过度设计

`decompose_input` 是为了演示"混合意图处理"能力而引入的，但在实际测试中带来的问题比解决的问题更多。

在医疗场景下，**安全性和体验一致性比功能完整性更重要**。宁可要求用户分步描述问题，也不能让系统输出割裂、混乱的回答。

### 大厂的实际做法

从公开产品（微信健康助手、京东健康等）的行为来看，主流方案是**单一意图路由**：整句话走 `classify_intent`，识别主意图，次要问题引导用户下一轮单独提问。

这样做的理由：
- 逻辑简单，不容易出 bug
- 对话连贯，用户体验一致
- 医疗场景出错代价高，保守优于激进

### 决策：方案 A — 单一意图路由 + 引导用户分步提问

去掉 `decompose_input`，整句话走单一意图路由。

当检测到混合输入时（`classify_intent` 置信度低或输入明显包含多类问题），系统主动引导：
> "您同时提到了就诊建议和用药问题，我先帮您解答就诊科室，用药问题请在下一条消息中单独提问。"

**为什么不选方案 B（真正的并行处理）：**
- 方案 B 需要 API 层 + 前端 UI 支持（独立回答块），CLI 阶段无法实现
- 在没有前端的情况下强行实现并行，只会得到割裂的线性输出
- 等 FastAPI + 前端完成后，方案 B 是自然的升级路径

**这个决策在面试中是成熟的产品判断，不是妥协：**
> "我们实现并测试了混合输入拆分方案（decompose_input），发现在 CLI 架构下存在追问异步性与顺序执行的根本矛盾，且两个完整问题并存时主从判断不稳定。综合评估后，当前阶段选择单一意图路由 + 引导分步提问，后续 API 层完成后再升级为前端并行回答块。这是根据当前架构能力做出的合理取舍，不是技术能力的限制。"

- 保留 `classify_intent()` 不变
