**TypedDict = 给字典写一份“字段说明书”**

普通 Python 字典长这样：

```
state = {    "user_input": "我想预约医生",    "step": 1,    "messages": [] } 
```

问题是：字典太自由了，谁都可以乱塞字段，或者把类型写错。

TypedDict 就是告诉 Python 和编辑器：

> 这个字典应该有哪些字段，每个字段应该是什么类型。

例如：

```
from typing import TypedDict class GraphState(TypedDict):    user_input: str    step: int    messages: list[dict] 
```

意思是：GraphState 这个状态字典里，应该有：

```
{    "user_input": 字符串,    "step": 数字,    "messages": 列表 } 
```

它本质上还是普通 dict，只是多了一层类型提示，方便检查错误。

需要注意一点：**TypedDict 本身不真正“保证” JSON 可序列化**，更准确地说是：如果你在 TypedDict 里只定义 str、int、list、dict、bool、None 这些 JSON 友好的类型，那么这个 state 就适合保存成 JSON。

不适合放进去的东西比如：

```
file = open("a.txt") db_connection = ... some_python_object = MyClass() 
```

这些通常不能直接 JSON 序列化。

**Checkpoint = 存档 / 保存进度**

checkpoint 可以理解成游戏里的“存档点”。

LangGraph 是一个图流程，比如：

```
用户输入 -> 分析意图 -> 查询信息 -> 生成回复 
```

每个节点都会读写同一个 state。

如果中途暂停了，比如：

- 等用户确认
- 人工审核
- 程序重启
- 多轮对话要继续
- 某一步执行中断

checkpoint 会把当前的 state 保存下来。

下次继续时，就不用从头开始，而是从之前保存的位置恢复。

------

**它们两个的关系**

一句话：

> TypedDict 负责定义“状态长什么样”，checkpoint 负责把这个状态保存起来，之后恢复。

比如 LangGraph 里的状态：

```
class State(TypedDict):    messages: list[dict]    user_id: str    current_step: str 
```

执行过程中可能变成：

```
{    "messages": [{"role": "user", "content": "我想挂号"}],    "user_id": "u123",    "current_step": "ask_department" } 
```

checkpoint 就会把这份状态保存起来。

下次用户继续说：

```
我要看皮肤科 
```

LangGraph 可以通过 checkpoint 找回之前的状态，知道这个用户之前正在“挂号流程”里，而不是重新开始。

------

你可以记成：

```
TypedDict：状态表结构 Checkpoint：状态存档 
```

在 LangGraph 里，“跨节点字段定义在此”通常就是指：所有节点之间共享的 state 字段，都统一写在这个 TypedDict 里。这样整个流程更清楚，也更容易中断后恢复。







这里的“本轮图 END”，不是说整个预诊流程结束，也不是患者会话结束。

它的意思是：**当前这一次图执行结束，把控制权交还给外部系统/用户。**

也就是：

```
用户发来一句话 → 图跑一轮 → 如果还缺信息，inquirer 生成 FOLLOW_UP → 图到 END → API 返回：请问患者 xxx？ → 等患者下一句话 → 下一轮重新从 intake 开始 
```

关键点是：**FOLLOW_UP 之后，图内没有新的患者信息。**

所以如果 inquirer 问完后又在同一轮图里回到 intake、monitor、controller，它拿到的还是旧 state。患者还没回答，缺失槽位不会 magically 被填上。这样继续跑只会出现几类问题：

1. **重复追问**
   controller 看到缺口还在，又让 prompter 生成下一问，可能一轮里问出多条问题。
2. **假装得到了答案**
   后面的节点为了继续推进，可能会基于缺失信息做推断，增加幻觉风险。
3. **调用不返回**
   对 API/前端来说，用户需要马上看到问题。如果图内部一直循环，外部系统拿不到 FOLLOW_UP 事件。
4. **状态边界混乱**
   “问了什么”“等哪个答案”“下一轮该合并到哪个槽位”会变得不清楚。

所以 END 的真实语义更像：

```
yield / pause / await user input 
```

不是 clinical 意义上的“完成”，而是工程上的“这一轮该停了”。

节点往返当然不是绝对不行。比如：

```
clinical → intake → clinical 
```

这种最多 back-loop 一次是合理的，因为它是在**同一批已有事实**里做补充抽取、结构化修正或重新评估，不依赖新的用户输入。

但：

```
inquirer → intake 
```

通常不合适，因为 inquirer 已经把问题抛给用户了，下一步必须等外部输入回来。

更准确地说，你的图可以理解为两层循环：

```
外层：多轮对话循环，由用户输入驱动 内层：单轮图执行，在一次输入内完成推理 
```

END 结束的是**内层单轮图执行**。外层会话还活着，预诊档案也继续保存。下一次患者回答后，再启动新一轮：

```
患者回答 → intake 抽取并合并新事实 → monitor → controller → ... 
```

所以这句可以改得更清楚：

> inquirer：emit FOLLOW_UP，并结束当前图执行，等待患者下一轮输入；会话状态保持为 awaiting_user。

这样就不会把 END 误解成“整个流程终止”。
